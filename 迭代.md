# ES 导入任务限时运行与断点续跑迭代方案

## 1. 背景

当前 ES 导入服务已经可以在较短时间内完成单表 T+1 同步。随着后续表数量增加、单表数据量增长到千万级甚至更高，导入任务可能因为数据量、网络、MySQL 查询耗时或 ES 写入波动导致执行时间明显拉长。

为了避免导入任务长时间占用 MySQL 资源，需要增加“单次任务最长运行时间”控制。到达时间上限后，服务停止继续读取 MySQL，但已经读取到内存队列的数据继续写入 ES，最后记录断点，等待下一次调度继续执行。

本次迭代不接入额外数据库，任务状态与断点信息统一记录在 ES 索引中。

## 2. 目标

- 每张表、每天的数据导入任务先落库，再由调度器执行。
- 单个任务支持最大运行时间配置，例如每天最多运行 8 小时。
- 到达 deadline 后停止继续读取 MySQL，避免继续占用数据库。
- 已经读取出的数据必须继续写入 ES，避免浪费查询结果。
- 任务未完整完成时，不切换 alias，线上查询继续使用上一版完整索引。
- 第二天或下一轮调度可以从断点继续执行未完成任务。
- 不引入额外数据库，使用 ES 任务索引 `sano_import_task` 保存任务状态。

## 3. 核心原则

### 3.1 不强制 kill 线程

不采用强制中断线程的方式结束任务。强制中断容易造成 JDBC 连接状态不确定、Bulk 请求结果不明确、断点与实际写入进度不一致。

正确方式是：

```text
到达 deadline
  -> 停止发起下一次 MySQL 查询
  -> 通知 Bulk worker 收尾
  -> 等待已入队数据写入 ES
  -> 更新任务状态为 TIMEOUT_PARTIAL
  -> 不切换 alias
```

### 3.2 只在查询下一页前检查 deadline

deadline 检查只放在每次准备查询下一页 MySQL 前。

如果一页数据已经从 MySQL 查询出来，就必须继续写入 ES。因为此时已经不再占用数据库资源，丢弃查询结果反而会浪费本次查询成本，并增加断点处理复杂度。

### 3.3 只有完整成功才切换 alias

只有任务状态达到 `SUCCESS`，才允许执行 refresh 和 switch alias。

如果任务状态是 `TIMEOUT_PARTIAL` 或 `FAILED`，说明新索引不完整，不能切换 alias。这样可以保证线上查询始终指向上一版完整索引。

### 3.4 断点以成功写入 ES 为准

断点字段使用 `last_success_id`，表示已确认成功写入 ES 的最大 MySQL 主键 ID。

续跑时从 `last_success_id` 之后继续读取。由于 ES 文档 `_id` 使用 MySQL 原始 ID，即使边界附近出现少量重复写入，也只会覆盖同一个文档，不会产生重复数据。

## 4. ES 任务索引设计

任务索引固定为：

```text
sano_import_task
```

该索引用于保存导入任务状态，不参与业务 alias 切换，也不跟随业务索引清理策略删除。

### 4.1 文档 ID

任务文档 `_id` 建议使用：

```text
{table_name}_{import_date}
```

示例：

```text
sano_wallet_coin_record_20260630
```

同一张表同一天只有一条任务记录。每天定时任务创建时使用 ES `create` 语义写入，如果文档已存在，则忽略。

### 4.2 文档结构

```json
{
  "table_name": "sano_wallet_coin_record",
  "index_alias": "sano_wallet_coin_record",
  "index_name": "sano_wallet_coin_record_20260630",
  "import_date": "20260630",
  "status": "PENDING",
  "last_success_id": 0,
  "total_count": 0,
  "success_count": 0,
  "failed_count": 0,
  "run_count": 0,
  "last_error": "",
  "started_at": "2026-07-01T02:00:00",
  "finished_at": null,
  "created_at": "2026-07-01T02:00:00",
  "updated_at": "2026-07-01T02:00:00"
}
```

### 4.3 字段说明

| 字段 | 说明 |
| --- | --- |
| `table_name` | MySQL 源表名 |
| `index_alias` | 业务查询使用的 ES alias |
| `index_name` | 本次任务写入的真实 ES 索引 |
| `import_date` | 导入业务日期，格式 `yyyyMMdd` |
| `status` | 任务状态 |
| `last_success_id` | 已确认成功写入 ES 的最大 MySQL ID |
| `total_count` | 本任务源端总数 |
| `success_count` | 已成功写入 ES 的数量 |
| `failed_count` | 写入失败数量 |
| `run_count` | 任务执行次数 |
| `last_error` | 最近一次失败原因 |
| `started_at` | 最近一次开始时间 |
| `finished_at` | 完成时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

## 5. 任务状态设计

状态枚举建议如下：

```text
PENDING          待执行
RUNNING          执行中
TIMEOUT_PARTIAL  超时暂停，可续跑
SUCCESS          完成，已切换 alias
FAILED           失败
CANCELLED        人工取消
```

推荐状态流转：

```text
PENDING -> RUNNING -> SUCCESS
PENDING -> RUNNING -> TIMEOUT_PARTIAL
TIMEOUT_PARTIAL -> RUNNING -> SUCCESS
RUNNING -> FAILED
RUNNING -> TIMEOUT_PARTIAL
```

## 6. 定时任务生成流程

每天定时触发时，不直接执行导入，而是先生成任务记录。

```text
1. 计算需要导入的业务日期，例如昨天 yyyyMMdd。
2. 遍历配置中的所有导入表。
3. 为每张表生成一个任务文档 ID。
4. 使用 ES create 语义写入 sano_import_task。
5. 如果任务已存在，忽略本次创建。
6. 调度器从任务索引中选择未完成任务执行。
```

这样即使前一天任务没有执行完，今天的新任务也会先记录为 `PENDING`，不会丢失。

## 7. 调度执行顺序

调度器每次从 ES 查询一条最早未完成任务：

```text
status IN (PENDING, TIMEOUT_PARTIAL)
ORDER BY import_date ASC, created_at ASC
LIMIT 1
```

执行顺序为：

```text
优先补历史未完成任务
再执行新日期任务
```

当前是单实例部署，可以先依赖应用内运行锁防止重复执行。后续如果升级为多实例，可基于 ES `seq_no` 和 `primary_term` 做乐观锁抢占。

## 8. 导入执行流程

```text
开始执行任务
  -> 将任务状态更新为 RUNNING
  -> 计算 deadline_millis
  -> 判断任务是否为续跑
  -> PENDING：创建新索引，从 last_success_id=0 开始
  -> TIMEOUT_PARTIAL：复用已有索引，从 last_success_id 继续
  -> 每次查询下一页 MySQL 前检查 deadline
  -> 未超时：读取 MySQL，写入 Bulk 队列
  -> Bulk 成功后推进 last_success_id
  -> 全部读取完成：refresh index，switch alias，状态改为 SUCCESS
  -> 到达 deadline：停止读取 MySQL，等待队列写完，状态改为 TIMEOUT_PARTIAL
  -> 发生异常：状态改为 FAILED
```

## 9. deadline 处理流程

任务开始时计算：

```text
deadline_millis = start_time + max_run_hours
```

每次准备查询下一页 MySQL 前执行：

```text
if now >= deadline_millis:
    stop reading mysql
    finish current bulk queue
    update task status to TIMEOUT_PARTIAL
    do not switch alias
```

deadline 检查只是一次数值比较，性能影响可以忽略。即使未来 1000 万数据约 1000 个批次，也只是约 1000 次时间比较。

## 10. alias 切换规则

```text
SUCCESS          允许 switch alias
TIMEOUT_PARTIAL  不允许 switch alias
FAILED           不允许 switch alias
CANCELLED        不允许 switch alias
```

如果任务未完整成功，新索引只是半成品索引，必须保留旧 alias 指向上一版完整索引。

## 11. 配置建议

```yaml
sano:
  es:
    import:
      max-run-hours: 8
      drain-timeout-minutes: 30
      page-query-timeout-seconds: 120
      resume-enabled: true
```

字段说明：

| 配置 | 说明 |
| --- | --- |
| `max-run-hours` | 单个导入任务最大运行小时数 |
| `drain-timeout-minutes` | 停止读取后等待 Bulk 队列写完的最长时间 |
| `page-query-timeout-seconds` | 单次 MySQL 查询超时时间 |
| `resume-enabled` | 是否允许从断点续跑 |

## 12. 关键边界

### 12.1 ES 不可用

任务状态保存在 ES 中，如果 ES 不可用，则任务状态也不可用。但本服务的目标就是写入 ES，ES 不可用时导入本身也无法进行，因此该风险可以接受。

### 12.2 任务索引不能被业务清理

`sano_import_task` 是系统任务索引，不能使用业务索引的历史清理逻辑。建议长期保留，或单独设置保留策略。

### 12.3 失败任务处理

`FAILED` 状态默认不自动继续，避免反复失败造成资源浪费。后续可增加人工接口，将失败任务重置为 `PENDING` 或 `TIMEOUT_PARTIAL` 后再执行。

### 12.4 RUNNING 残留

如果服务进程异常退出，可能留下 `RUNNING` 状态任务。后续可增加启动修复逻辑：

```text
启动时扫描 `updated_at` 超过阈值的 RUNNING 任务
  -> 改为 TIMEOUT_PARTIAL 或 FAILED
```

单机部署下建议先实现简单版本，避免一次迭代引入过多复杂度。

## 13. 推荐迭代拆分

### 第一阶段：任务索引与任务模型

- 新增 `sano_import_task` mapping。
- 新增任务模型 `EsImportTask`。
- 新增任务状态枚举。
- 新增任务服务 `EsImportTaskService`。
- 支持创建任务、查询待执行任务、更新任务状态。

### 第二阶段：调度入口改造

- 定时任务先创建 `PENDING` 任务。
- 调度器按日期顺序选择 `PENDING`、`TIMEOUT_PARTIAL` 任务。
- 原有手动接口可继续保留，但内部也应复用任务执行逻辑。

### 第三阶段：deadline 与收尾

- 新增最大运行时间配置。
- `JdbcDataReader` 每次查询下一页前检查 deadline。
- 超时后停止读取，等待 Bulk 队列写完。
- 超时任务更新为 `TIMEOUT_PARTIAL`，不切换 alias。

### 第四阶段：断点续跑

- Bulk 成功后推进 `last_success_id`。
- `TIMEOUT_PARTIAL` 任务复用已有索引。
- 从 `last_success_id` 后继续读取 MySQL。
- 完整完成后 refresh、switch alias、更新为 `SUCCESS`。

### 第五阶段：运维增强

- 增加任务状态查询接口。
- 增加失败任务重置接口。
- 增加 RUNNING 残留任务修复逻辑。
- 增加任务索引保留策略。

## 14. 最终效果

完成该迭代后，导入服务将具备：

- 多表多天任务可排队执行。
- 每日任务不会因前一天未完成而丢失。
- 超时后不继续占用 MySQL。
- 已读取数据会继续写完 ES。
- 未完成索引不会污染线上 alias。
- 后续调度可以从断点继续。
- 不依赖额外数据库，部署复杂度保持较低。
