# ES 与 es-server Docker 部署说明

本文档说明 `elasticsearch8` 与 `es-server` 的推荐部署方式：两个 compose 文件独立维护，但加入同一个 Docker 外部网络，通过容器服务名通信。

## 一、部署结构

目录结构示例：

```text
C:\work\opts\sano\code\datahub
├── elasticsearch8
│   └── docker-compose-elasticsearch.yml
└── es-server
    ├── Dockerfile
    ├── docker-compose.yml
    ├── docker-build.bat
    ├── docker-build.sh
    └── deploy-es-server.sh
```

推荐通信方式：

```text
es-server 容器 -> sano-net -> elasticsearch8:9200
```

说明：

- `9200` 是 ES 容器内部端口，容器间通信使用它。
- `9201` 是宿主机映射端口，主要用于宿主机或外部工具访问 ES。
- `host.docker.internal` 只适合容器访问宿主机服务，本部署不需要。

## 二、检查与创建共享网络

本项目使用外部 Docker 网络 `sano-net` 让 ES 与 es-server 通过容器名互相访问。

先查看当前 Docker 网络：

```powershell
docker network ls
```

检查 `sano-net` 是否已存在：

```powershell
docker network ls | findstr sano-net
```

如果没有输出，创建网络：

```powershell
docker network create sano-net
```

如果网络已存在，会提示重复创建，忽略即可。

查看网络详情：

```powershell
docker network inspect sano-net
```

重点检查：

- `Name` 是否为 `sano-net`。
- `Driver` 是否为 `bridge`。
- `Containers` 中是否包含 `elasticsearch8` 和 `es-server`。

如果服务已经启动，也可以用下面命令快速确认容器是否加入网络：

```powershell
docker inspect elasticsearch8 --format "{{json .NetworkSettings.Networks}}"
docker inspect sano-es-server --format "{{json .NetworkSettings.Networks}}"
```

输出中应能看到 `sano-net`。

容器间连通性测试：

```powershell
docker exec es-server curl -I http://elasticsearch8:9200
```

如果返回 `401 Unauthorized`，通常是正常的，说明网络已通，只是 ES 开启了账号密码认证。

## 三、启动 Elasticsearch

进入 ES 目录：

```powershell
cd C:\work\opts\sano\code\datahub\elasticsearch8
```

启动：

```powershell
docker-compose -f docker-compose-elasticsearch.yml -p elasticsearch8 up -d
```

查看状态：

```powershell
docker ps
docker logs -f elasticsearch8
```

宿主机访问 ES：

```powershell
curl http://127.0.0.1:9201
```

## 四、构建 es-server 镜像

进入 es-server 目录：

```powershell
cd C:\work\opts\sano\code\datahub\es-server
```

本地构建镜像：

```powershell
.\docker-build.bat
```

构建脚本会先执行 `mvn clean package`，再制作 Docker 镜像。`clean` 用于清理旧 `.class` 文件，避免已删除的历史类被误打进 jar。

构建指定版本：

```powershell
.\docker-build.bat v1.0.0
```

构建并推送多架构镜像：

```powershell
.\docker-build.bat v1.0.0 --push
```

当指定版本号并使用 `--push` 时，脚本会同时推送两个 tag：

```text
lxw13000/sano-es-server:v1.0.0
lxw13000/sano-es-server:latest
```

例如当前已有 `v1.0.0`、`v1.0.1`，执行：

```powershell
.\docker-build.bat v1.0.2 --push
```

Docker Hub 上会更新：

```text
v1.0.2
latest
```

说明：正式服务器如果是 ARM 架构，必须使用 `--push` 推送多架构镜像。只执行 `.\docker-build.bat v1.0.0` 只会生成当前机器架构的本地镜像，不会让 Docker Hub 上的 `v1.0.0` 同时支持 amd64 和 arm64。

检查 Docker Hub 上某个 tag 是否包含多架构 manifest：

```powershell
docker buildx imagetools inspect lxw13000/sano-es-server:v1.0.0
```

输出中应同时看到类似：

```text
Platform: linux/amd64
Platform: linux/arm64
```

如果 ARM 服务器启动时报：

```text
no matching manifest for linux/arm64/v8 in the manifest list entries
```

说明该 tag 没有 arm64 镜像，需要重新执行：

```powershell
.\docker-build.bat v1.0.0 --push
```

Linux/macOS 可使用：

```bash
./docker-build.sh
```

## 五、启动 es-server

首次启动前，先创建并授权日志目录：

```bash
cd /path/to/es-server
mkdir -p logs
sudo chown -R 10001:10001 logs
chmod 755 logs
```

说明：es-server 容器内使用非 root 用户运行，固定 UID/GID 为 `10001:10001`。如果宿主机 `logs` 目录属于其他用户且没有写权限，应用会无法创建 `yyyyMMdd` 日期目录，并出现类似：

```text
Failed to create parent directories for [/app/logs/es-server/yyyyMMdd/es-server.log]
```

启动：

```powershell
docker-compose -f docker-compose.yml -p sano-es-server up -d
```

默认使用镜像：

```text
lxw13000/sano-es-server:latest
```

如果要部署指定版本，例如 `v1.0.0`，可在服务器上使用环境变量指定：

```bash
ES_SERVER_IMAGE_TAG=v1.0.0 docker-compose -f docker-compose.yml -p sano-es-server up -d
```

PowerShell 中使用：

```powershell
$env:ES_SERVER_IMAGE_TAG="v1.0.0"
docker-compose -f docker-compose.yml -p sano-es-server up -d
```

查看日志：

```powershell
docker logs -f sano-es-server
```

访问服务：

```powershell
curl http://127.0.0.1:8002
```

## 六、使用运维脚本部署 es-server

Linux 服务器推荐使用 `deploy-es-server.sh` 发布 es-server。脚本会执行：

- 停止 compose 服务。
- 删除旧容器。
- 删除本地 `lxw13000/sano-es-server` 相关旧镜像。
- 拉取指定版本镜像。
- 重新启动服务。
- 检查容器是否真实启动成功，避免把反复重启误判为成功。

进入 es-server 目录：

```bash
cd /path/to/es-server
chmod +x deploy-es-server.sh
```

部署默认 `latest`：

```bash
./deploy-es-server.sh
```

部署指定版本：

```bash
ES_SERVER_IMAGE_TAG=v1.0.0 ./deploy-es-server.sh
```

如果指定版本不存在或拉取失败，脚本会自动回退到：

```text
lxw13000/sano-es-server:latest
```

脚本默认配置：

```text
PROJECT_NAME=sano-es-server
SERVICE_NAME=es-server
CONTAINER_NAME=sano-es-server
ES_SERVER_IMAGE=lxw13000/sano-es-server
ES_SERVER_IMAGE_TAG=latest
START_TIMEOUT=180
STABLE_SECONDS=20
```

真实启动成功判断：

- 如果镜像定义了 Docker `HEALTHCHECK`，必须等到 `healthy`。
- 如果没有健康检查，会检查容器稳定运行一段时间且 `/health` 可访问。
- 如果容器退出、OOM、处于 restarting、或启动后重启次数变化，会判定为失败并打印最近日志。

## 七、关键配置说明

`es-server/docker-compose.yml` 中：

```yaml
ES_URIS: ${ES_URIS:-elasticsearch8:9200}
```

这里的 `elasticsearch8` 来自 ES compose 中的服务名和容器名：

```yaml
services:
  elasticsearch8:
    container_name: elasticsearch8
```

只要两个服务都加入 `sano-net`，es-server 就可以直接访问：

```text
elasticsearch8:9200
```

## 八、常用维护命令

停止 es-server：

```powershell
cd C:\work\opts\sano\code\datahub\es-server
docker-compose -f docker-compose.yml -p sano-es-server down
```

停止 ES：

```powershell
cd C:\work\opts\sano\code\datahub\elasticsearch8
docker-compose -f docker-compose-elasticsearch.yml -p sano-es down
```

查看共享网络内的容器：

```powershell
docker network inspect sano-net
```

检查容器是否加入共享网络：

```powershell
docker inspect elasticsearch8 --format "{{json .NetworkSettings.Networks}}"
docker inspect sano-es-server --format "{{json .NetworkSettings.Networks}}"
```

网络异常时，先停止相关容器，再重新启动：

```powershell
cd C:\work\opts\sano\code\datahub\es-server
docker-compose -f docker-compose.yml -p sano-es-server down

cd C:\work\opts\sano\code\datahub\elasticsearch8
docker-compose -f docker-compose-elasticsearch.yml -p sano-es down

docker network create sano-net
```

如果确认没有容器正在使用该网络，可以删除后重建：

```powershell
docker network rm sano-net
docker network create sano-net
```

注意：只有在 `docker network inspect sano-net` 中 `Containers` 为空时，才建议删除网络。

重建 es-server：

```powershell
cd C:\work\opts\sano\code\datahub\es-server
.\docker-build.bat
docker-compose -f docker-compose.yml -p sano-es-server up -d
```

## 九、部署建议

- 生产环境建议先启动 ES，确认 ES 健康后再启动 es-server。
- ES 数据目录、日志目录、备份目录由 `elasticsearch8` 目录下的挂载目录维护。
- es-server 日志挂载到 `es-server/logs`，便于排查导入任务。
- es-server 容器内应用用户固定为 `10001:10001`，宿主机 `logs` 目录需要提前授权。
- Docker Desktop 不建议为本项目强制配置代理；能直连镜像源时保持代理为空更稳定。
